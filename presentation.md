# ニコニコライエジョフ

LLMでニコニコ動画のコメントをスマートにフィルターする

---

## 1. ニコニコ動画のコメント、好きですか？

ニコニコ動画の真髄はコメント機能。動画の上を流れるコメントが、一人で観る動画を「みんなで観ている」体験に変える。

しかし、そのコメント機能は良い方向に振れることが多い一方で、悪い方向に振れるときもある。

- 全然関係ない話題で喧嘩が始まって画面が埋め尽くされる
- コントラバーシャルな話題に触れて紛糾している
- 特定のミームやネタの連投で本来のコメントが読めなくなる

---

## 2. 既存のNG機能の限界

こういった場合のためにコメントNGやユーザーNG機能はあるにはある。

しかし：

- あらゆる話題について事前にNGワードを設定するのは**実質的に不可能**
- 文脈によって問題になるコメントは、単語単位のNGでは捕捉できない
- ユーザーNGは事後対応でしかなく、新しいユーザーには効かない
- 設定の手間が動画を楽しむ体験を損なう

**ルールベースのフィルターでは、この問題を根本的に解決できない。**

---

## 3. そこで作ったのが「ニコニコライエジョフ」

（名前は5秒で考えました）

LLMモデルを使ってコメントをフィルターするブラウザ拡張。

なぜLLMか？

- 文脈を加味した上でコメントを判断できる
- 「喧嘩」「荒らし」「ネタバレ」のような抽象的な概念を理解できる
- ユーザーが自然言語で「こういうコメントを消して」と指示できる
- 新しいパターンの荒らしにも対応できる

---

## 4. 機能紹介

### 機能1: AIによるコメント欄の治安評価

チャット画面を開いて「分析開始」ボタンを押すと、AI人民委員がコメント欄の治安を天気予報形式で評価する。

- ☀️ 快晴 — 良好、フィルター不要
- ☁️ 曇り — やや荒れ、フィルター推奨
- ⛈️ 雷雨 — かなり荒れている、粛清必要

問題がある場合はフィルター提案がボタンとして表示され、選択して即実行できる。

### 機能2: コメントの内容についてAIと会話する

チャットインターフェースで動画のコメントについて質問・指示できる。

- 「この動画で荒れてるコメントある？」
- 「政治的な話題のコメントだけ非表示にして」
- 「英語コメントを日本語に翻訳して」

指示するとAIが即座にツールコールで実行する。確認ステップなしの即時粛清。

### 機能3: マルチプロバイダー対応

3つのLLMプロバイダーから好きなモデルを選べる:

| プロバイダー | モデル例 | 特徴 |
|---|---|---|
| Anthropic (Claude) | Haiku 4.5, Sonnet 4.6, Opus 4.6 | tool use の精度が高い |
| Google (Gemini) | Flash-Lite, 3.5 Flash, 3.1 Pro | 低コストで高速 |
| OpenAI (GPT) | GPT-5.4 nano/mini/5.4, GPT-5.5 | 幅広いモデル選択肢 |

モデル切り替えは設定タブから即座に可能。コストタブで利用量を比較できる。

### 機能4: コスト追跡

利用トークン数・コスト（USD / 円換算）を自動記録。モデルごと・動画ごとの履歴が確認できる。

## 8. 技術スタック

### 全体構成

```
ブラウザ (ニコニコ動画)
  └── Tampermonkey ユーザースクリプト (単一ファイル)
        ├── UI層: ボタン注入 + チャットモーダル (タブ: チャット / 設定 / コスト)
        ├── コメント操作層: React Fiber 走査 → Immer ストア / PixiJS レンダラー書き換え
        └── LLM層: マルチプロバイダー API (tool use / Function Calling)
              ↕ GM_xmlhttpRequest (CORS回避)
        Anthropic API / Gemini API / OpenAI API
```

### ニコニコ動画プレイヤーの内部構造（解析済み）

| 要素 | 技術 | 役割 |
|---|---|---|
| 状態管理 | Immer ベースのカスタムストア (クラス `se`) | コメントデータの保持。`comments` / `filteredComments` |
| 状態保持 | Jotai アトム | ストアインスタンスを React ツリー内で保持 |
| 描画 | PixiJS (Canvas2D) | 弾幕コメントのレンダリング。クラス `Qy` |
| UIフレームワーク | React + Panda CSS | プレイヤーUI全般 |

### コメント書き換えの仕組み

React Fiber ツリーを `#root` から BFS 走査し、ストアとレンダラーのインスタンスを直接取得する。

```
#root.__reactContainer$ → Fiber BFS
  → ストア検出: update() + current() + subscribe() を持ち、current().comments が存在
  → プレイヤー検出: commentRenderer プロパティを持つオブジェクト
```

書き換えは3段階:

1. `store.update(draft => {...})` で Immer draft 経由で `comments` + `filteredComments` を改変
2. レンダラーの `layerProcessorList[].stagingChatManager.chatList[].content` を直接書き換え
3. `renderer.refreshCommentsByTarget()` で再描画

### LLM連携: マルチプロバイダー tool use

ユーザーの自然言語の指示をLLMが解釈し、tool use / Function Calling で具体的な操作に変換する。

```
ユーザー: 「荒らしコメントを消して、英語コメントは翻訳して」
    ↓
LLM → tool_use: hide_comments({ids: [...]})
    → tool_use: replace_comments({replacements: [{id, new_body}, ...]})
    ↓
ユーザースクリプトが tool call を実行 → ストア + レンダラーに反映
```

定義しているツール:

| ツール名 | 説明 |
|---|---|
| `hide_comments` | 指定IDのコメントを非表示にする |
| `replace_comments` | 指定IDのコメントの本文を書き換える（翻訳・検閲など） |

各プロバイダーのツール定義形式に自動変換:
- Anthropic: `input_schema` (そのまま)
- Gemini: `functionDeclarations` + `parameters`
- OpenAI: `{ type: 'function', function: { parameters } }`

レスポンスは共通形式に正規化されるため、ツール実行コードはプロバイダーに依存しない。

テキスト出力で JSON を返させるのではなく tool use を使う理由:

- **構造が保証される** — パース失敗がない
- **複合操作** — 1回のレスポンスで複数ツールを呼べる（非表示と翻訳を同時に）
- **説明と操作の分離** — 「3件検出しました」というテキストと、実際の操作が別チャネル

### API呼び出し: GM_xmlhttpRequest

通常のブラウザ `fetch` では外部 API への呼び出しが CORS でブロックされる。
Tampermonkey の `GM_xmlhttpRequest` はブラウザの同一オリジンポリシーを迂回できるため、
SDKなしで直接 API を呼べる。

### デザイン: ニコニコ公式のカラートークン

ニコニコ動画の CSS から Panda CSS のデザイントークンを抽出し、UI に適用。
プレイヤーのダークモードに完全に馴染む。

## 9. どうやって作った？

### リバースエンジニアリング

ニコニコ動画のプレイヤーはReact + PixiJSで構築されており、コードは難読化（minify）されている。

**自分では一切リバエンしていない。**

1. ChromeのDevToolsからHARファイル（ネットワーク通信の記録）をエクスポート
2. そのままClaude Codeに渡して「コメントシステムの内部構造を調べて」と指示
3. Reactの仮想DOM、Immerの状態管理、PixiJSの描画パイプラインまで全部解析された
4. その結果をもとに「コメントを書き換えるスクリプトを作って」と指示
5. React Fiberツリーを走査してストアとレンダラーに直接アクセスするコードが生成された

難読化されたコードの解読、内部APIの特定、動作するPoCの作成まで、すべてClaude Codeが行った。

### 成果物

- **36KBの内部構造ドキュメント** — API仕様、データフロー、minifiedシンボルの対応表
- **動作するユーザースクリプト** — Tampermonkeyで動く単一ファイル
- **ニコニコのデザイントークン** — 公式UIに馴染むカラースキーム

## 10. 50年後のフィルター

### コンテンツは無限に増え続ける

2025年現在、ニコニコ動画には数千万本の動画と数十億のコメントがある。
50年後、コンテンツ量は桁違いに膨れ上がり、参加者も地球規模を超えているかもしれない。

そのとき、「NGワード」や「ユーザーブロック」で対処できるだろうか？

### ルールベースの限界はすでに見えている

| 時代 | フィルター手法 | 限界 |
|---|---|---|
| 2000年代 | NGワード | 文脈を無視、回避が容易 |
| 2010年代 | 機械学習分類器 | 学習データに依存、新しいパターンに弱い |
| 2020年代 | LLM tool use (今ここ) | 文脈を理解し、自然言語で指示できる |
| 2070年代？ | ？ | ？ |

### 50年後、こうなっているかもしれない

- **パーソナライズされたフィルター**: 全員に同じコメントが見える必要はない。各ユーザーが自分の好みに合った「人民委員」を持ち、自分だけのコメント体験を構築する
- **多言語・多文化の同時対応**: 100言語のコメントが同時に流れ、各ユーザーの言語と文化圏に合わせてリアルタイムに翻訳・フィルターされる
- **文化の文脈理解**: 弾幕や空耳のような文化的コメントを潰さずに守る知性。荒らしと文化の境界線を引けるのは、文脈を理解するAIだけ
- **コスト問題の解消**: 今は1動画あたり数円のAPIコストがかかるが、50年後にはフィルターのコストはゼロに近づき、プラットフォームに標準搭載されているかもしれない

### ニコニコライエジョフは、その最初の一歩

LLMに「このコメント欄の治安を守って」と自然言語で頼めること自体が、
50年後には当たり前になっているフィルター体験の原型。

今日のプロトタイプが、50年後の標準機能の先取りかもしれない。

## 11. まとめ

- ニコニコのコメント体験を守るために、LLMでスマートにフィルターする
- ルールベースでは不可能だった「文脈を理解したフィルター」を実現
- 3つのLLMプロバイダーに対応、コストに応じてモデルを選べる
- リバエンからUI実装まで、Claude Codeとの協業で短期間で構築
